# 爆款拆解 - 内容分析双栏方案

## 一、需求概述

- **当前**：内容分析 Tab 为单栏表单（手动填标题/描述 + 开始分析），分析结果在右侧展示。
- **目标**：改为与「账号分析」一致的双栏布局——
  - **左侧**：当前爆款账号下的**内容列表**（从哪个账号切过来就显示该账号的全部内容），支持搜索、分页（每页 4 条）、新增/编辑/删除、每条提供「分析」操作。
  - **右侧**：选中内容的**分析结果**，与左侧列表联动（点击列表项或分析完成后展示对应结果）。

---

## 二、数据设计

### 2.1 已有表（无需新建）

| 表 | 说明 | 关键字段 |
|----|------|----------|
| `knowledge_accounts` | 爆款账号 | id, name, platform, url, current_data, analysis_result, ... |
| `knowledge_contents` | 账号下的内容 | id, **account_id**, title, content_url, content_type, source_type, content_data, **analysis_result**, created_at, updated_at |

- 内容与账号关联：`KnowledgeContent.account_id` → `KnowledgeAccount.id`，已有 `contents` 关系。
- 分析结果存储：`KnowledgeContent.analysis_result`（JSON），已有；分析完成后写入即可。

### 2.2 可选扩展（按需）

- 若后续需要「内容级分析历史/版本」：可增加 `knowledge_content_analysis_history` 表，当前阶段用 `analysis_result` 单字段即可。

---

## 三、后端 API 设计

### 3.1 内容列表（改造）

**接口**：`GET /api/knowledge/contents`

**现有**：支持 `account_id`、`content_type`、`source_type`，但**无分页、无搜索**，一次返回全部。

**改造**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| account_id | int | 是（在内容分析场景） | 只查该账号下的内容 |
| page | int | 否，默认 1 | 页码 |
| page_size | int | 否，默认 4 | 每页条数（前端约定 4） |
| search | string | 否 | 搜索：标题 title 或 content_data.description 模糊匹配 |

**响应**（与现有格式兼容，增加分页信息）：

```json
{
  "code": 200,
  "message": "获取成功",
  "data": {
    "items": [
      {
        "id": 1,
        "account_id": 10,
        "title": "标题",
        "content_url": "",
        "content_type": "video",
        "source_type": "manual",
        "content_data": { "description": "..." },
        "analysis_result": null,
        "created_at": "...",
        "updated_at": "..."
      }
    ],
    "total": 20,
    "page": 1,
    "page_size": 4,
    "total_pages": 5
  }
}
```

- 当未传 `account_id` 时，可保留原逻辑（返回全部内容，若数据量大建议仍加分页）。

### 3.2 内容 CRUD（已有，确认）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/knowledge/contents/<content_id> | 单条详情（含 analysis_result） |
| POST | /api/knowledge/contents | 新增（body: title, account_id, content_type, source_type, content_data） |
| PUT | /api/knowledge/contents/<content_id> | 更新 |
| DELETE | /api/knowledge/contents/<content_id> | 删除 |

新增时若从「内容分析」当前账号进入，前端固定带 `account_id = currentAccount.id`。

### 3.3 内容分析并回写（扩展）

**接口**：`POST /api/knowledge/analyze-with-account`（现有）

**扩展请求体**：增加可选字段 `content_id`（int）。

- 若传 `content_id`：
  - 从 `KnowledgeContent` 取该条记录的 title、content_data（如 description）、content_type、account_id。
  - 用对应账号信息 + 内容信息调用现有分析逻辑。
  - 分析成功后，将结果写入 `KnowledgeContent.analysis_result`，并返回与现有一致的 `data`。
- 若不传 `content_id`：保持当前行为（用 url + content_info 等），不写库。

这样「列表中点分析」可一次请求完成分析 + 落库，右侧用返回结果或再 GET 该 content 即可。

### 3.4 可选：仅分析并回写

若希望「分析」接口语义更清晰，可单独增加：

- `POST /api/knowledge/contents/<content_id>/analyze`  
  - 内部：取 content + 其 account，调用现有分析逻辑，写回 `analysis_result`，返回分析结果。

实现上可与 `analyze-with-account` 共用分析逻辑，仅多一个「按 content_id 取数」的入口。

## 四、前端设计

### 4.1 布局（参考账号分析 Tab）

- **内容分析 Tab** 内采用与「账号分析」一致的两栏：
  - 左侧一列：**当前账号信息条** + **内容列表区**（搜索 + 列表 + 分页）+ **新增按钮**。
  - 右侧一列：**分析结果区**（与现有 result-container 一致，展示 content 的 analysis_result）。

### 4.2 左侧：当前账号 + 内容列表

- **当前账号**：与现有一致，显示「当前选中的爆款账号」名称 + 「切换账号」按钮（切回账号分析或清空后重选）。从哪个账号切到内容分析，就认为 currentAccount 即该账号。
- **内容列表**：
  - 搜索：单输入框，placeholder「搜索标题/描述」，回车或按钮触发，请求 `GET /api/knowledge/contents?account_id=当前账号id&page=1&page_size=4&search=关键词`。
  - 列表项（每页 4 条）：卡片展示标题、类型、是否有分析结果（如角标或图标）。操作：**分析**、**编辑**、**删除**。
  - 分页：上一页 / 页码信息 / 下一页，与账号分析分页风格一致。
- **新增**：按钮「新增内容」，打开表单（Modal 或内联）：标题、内容类型、描述/文案等，保存时 `account_id` 固定为当前账号 id，成功后刷新列表并可选选中新项。

### 4.3 右侧：分析结果

- 选中某条内容（点击列表项或点击「分析」并完成）后：
  - 若该条已有 `analysis_result`：直接展示（与现有 `displayAnalysisResult` 一致）。
  - 若触发「分析」：先调分析接口（带 `content_id`），成功后写库并返回结果，再展示；可带 loading。
- 未选中或选中但无结果时：占位文案「请在左侧选择内容或进行分析」。

### 4.4 联动与状态

- 切换账号（在账号分析 Tab 选账号后切到内容分析）：左侧内容列表按新账号的 `account_id` 重新拉取，分页重置为 1，右侧可清空或提示「请选择内容」。
- 列表「编辑」：弹窗编辑标题/类型/描述等，保存后刷新列表；若当前选中为该条，可可选刷新右侧（若只改了文案未重新分析，可仍显示旧分析结果并提示「内容已更新，可重新分析」）。
- 列表「删除」：确认后调用 DELETE，成功后从列表移除；若删除的为当前选中，右侧清空。

### 4.5 与现有「分析队列」的关系

- 若内容分析仍走现有「分析队列」：可将「内容分析」任务与「账号分析」任务区分类型，队列只展示任务列表即可；列表中的「分析」按钮可视为提交一个内容分析任务，完成后通过轮询或 WebSocket 更新该条的 `analysis_result` 并在右侧展示。
- 若内容分析为同步调用（当前 analyze-with-account 为同步）：则点击「分析」即等待接口返回，成功后更新当前条并刷新右侧，无需队列。

## 五、其他考虑

1. **权限**：沿用现有知识库/爆款拆解权限，内容列表与内容分析接口与现有 `/api/knowledge/*` 一致，`@login_required` 即可。
2. **空状态**：当前账号下无内容时，列表展示「暂无内容，可点击新增」；未选账号时内容分析 Tab 可禁用（与现有逻辑一致）。
3. **搜索**：搜索建议对 `title` 与 `content_data->description`（或 content_data 内主文案字段）做模糊匹配；若 content_data 为 JSON，可用 DB 的 JSON 查询或应用层过滤。
4. **分析维度**：内容分析仍使用现有「分析维度管理」中启用的内容分析维度，无需变更。
5. **一键入库**：若现有有「从分析结果入库到规则库/素材库」的入口，保留在右侧结果区，仍针对当前选中的 content 的 analysis_result 操作即可。
